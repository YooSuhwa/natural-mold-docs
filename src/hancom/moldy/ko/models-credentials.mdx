Moldy는 모델 catalog와 자격증명을 분리합니다. 모델은 어떤 LLM을 선택할 수 있고 어떻게 표시되는지를 정의하며, 자격증명은 provider 호출에 필요한 비밀값을 제공합니다.

이 분리는 secret 값을 노출하지 않고 모델 구성을 감사 가능하게 유지합니다. 모델 호출이 실패하면 에이전트 지침을 바꾸기 전에 모델 catalog 항목, 선택된 자격증명, 런타임 범위를 먼저 확인하세요.

## 사용자 자격증명

<img className="moldy-screenshot" src="/images/hancom/moldy/ko/credentials.png" alt="Moldy 사용자 자격증명 목록 화면" />

사용자 자격증명은 **Settings > Credentials**에서 관리하며 개인 에이전트, 도구, MCP 서버, 모델 테스트, 모델 discovery에 사용됩니다. 백엔드 응답은 복호화된 secret 값을 반환하지 않고, 상태와 field key, OAuth 지원 여부, audit log, 테스트 결과를 제공합니다.

지원되는 credential definition은 백엔드 `credentials/definitions`에 등록됩니다. 현재 소스에는 OpenAI, Anthropic, Google GenAI, OpenRouter, Azure OpenAI, HTTP API key/basic/bearer, MCP OAuth2/secret, Google Workspace OAuth2, 검색/교통/공공 API 계열 정의가 포함됩니다.

## 시스템 자격증명

<img className="moldy-screenshot" src="/images/hancom/moldy/ko/system-credentials.png" alt="Moldy 시스템 자격증명 화면" />

시스템 자격증명은 운영자가 관리합니다. `is_system=true`로 저장되어 사용자 선택 목록에는 노출되지 않고, Builder, Assistant, 이미지 생성, System LLM 같은 제품 내부 흐름에 사용됩니다.

| 구분 | 사용자 자격증명 | 시스템 자격증명 |
| --- | --- | --- |
| 소유 범위 | 현재 사용자 | 운영자 관리 |
| 조회 권한 | 현재 사용자 | `super_user` |
| 사용처 | 사용자 에이전트, 도구, MCP 서버 | System LLM, Builder, Assistant, 이미지 생성 |
| UI 위치 | 자격증명 | 시스템 자격증명 |

## 모델 catalog

<img className="moldy-screenshot" src="/images/hancom/moldy/ko/models.png" alt="Moldy 모델 카탈로그 화면" />

**Settings > Models**는 provider, model name, display name, base URL, 가격, context window, modality, function calling, reasoning, visibility, default credential, health status, 가능한 경우 benchmark ranking data를 관리합니다.

모델 생성/수정/삭제는 운영자 권한이 필요합니다. 삭제는 해당 모델을 사용하는 에이전트가 있으면 거부됩니다.

모델 메타데이터는 런타임 선택, reporting, catalog sorting에 영향을 줍니다. 가격 메타데이터는 추정 비용 계산에 쓰이고, 기능 플래그는 tool-calling과 표시 판단에 영향을 주며, ranking은 알려진 public model 비교에 도움을 주고, visibility는 사용자가 선택할 수 있는 모델을 제어합니다.

## 모델 테스트와 발견

모델은 등록만으로 충분하지 않습니다. 호출 가능한 자격증명이 필요합니다. Moldy는 저장된 모델 테스트와 preview test, credential 기반 모델 discovery, health check를 제공합니다.

테스트 실패 시 확인 순서는 다음과 같습니다.

1. 모델 provider와 model name이 실제 제공자와 일치하는지 확인합니다.
2. 모델에 default credential이 있거나 테스트 요청에 credential을 지정했는지 확인합니다.
3. base URL이 필요한 제공자라면 URL이 올바른지 확인합니다.
4. 가격/기능 플래그는 호출 가능 여부와 별개로 catalog 표시와 사용량 계산에 영향을 줍니다.
5. discovery로 찾은 모델은 저장 전에 benchmark ranking과 ranking이 없는 fallback 표시를 확인합니다.

## 에이전트 모델 선택과 fallback

에이전트는 기본 모델과 fallback 모델 목록을 가질 수 있습니다. 채팅 런타임은 기본 모델을 먼저 사용하고, fallback 목록을 순서대로 해석합니다.

Fallback은 안정성을 위한 장치이지 권한 우회 수단이 아닙니다. 각 fallback 모델도 올바른 범위의 호출 가능한 자격증명이 필요하며, 운영 환경의 에이전트 워크플로에 의존하기 전에 테스트해야 합니다.

<Note>
fallback 모델이 있어도 해당 provider의 사용자 자격증명이 없으면 호출에 실패할 수 있습니다. 운영자 시스템 자격증명은 사용자 채팅 런타임의 일반 대체 수단이 아닙니다.
</Note>
