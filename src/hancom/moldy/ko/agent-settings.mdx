에이전트 설정 화면은 Moldy 에이전트의 제어 화면입니다. 이 화면에서 지침, 모델, 도구, MCP 도구, 스킬, 서브에이전트, 미들웨어, 메모리 동작, 오프너 질문, 스케줄, API 관련 설정, 표시 정보를 바꾸며, 대화 실행 시 런타임은 이 저장된 설정을 읽습니다.

이 구성은 `agents/[agentId]/settings` 화면과 `agent.visualSettings` 플로우에 실제로 구현되어 있습니다. 어떤 저장 필드가 에이전트 동작에 영향을 주고, 어떤 패널이 테스트/검토/표시용인지 설명할 때 이 문서를 기준으로 사용하세요.

<img className="moldy-screenshot" src="/images/hancom/moldy/ko/agent-settings.png" alt="Moldy 에이전트 설정 화면" />

## 설정 화면 구조

| 영역 | 실제로 조정하는 항목 |
| --- | --- |
| 기본 정보 | 이름, 설명, 시스템 지침 |
| 모델 | 기본 모델, temperature, top_p, max_tokens, fallback 모델 순서 |
| 연결 | 내장 도구, MCP 도구, 스킬 |
| 구성 요소 | 서브에이전트, 미들웨어 |
| 오른쪽 패널 | Fix, 테스트 채팅, 오프너 질문, 스케줄, 설정 작업, API 패널 |
| 비주얼 모드 | 노드 기반으로 에이전트, 도구, 스킬, 스케줄, 미들웨어 관계를 확인 |

폼 모드는 정확한 설정 변경에 적합합니다. 비주얼 모드는 에이전트, 모델, 확장 기능, 스케줄의 관계를 확인해 런타임 구성을 바꾸지 않고 누락된 연결을 찾을 때 적합합니다.

## 지침 작성

지침은 에이전트의 역할, 금지사항, 응답 형식, 도구 사용 기준을 정의합니다. Moldy는 `system_prompt`를 에이전트 설정으로 저장하고, 대화 실행 시 에이전트 런타임 구성에 포함합니다.

좋은 지침에는 다음이 들어갑니다.

- 에이전트가 맡을 업무 범위
- 사용자가 기대하는 답변 톤과 형식
- 도구를 호출해야 하는 조건
- 모르는 정보가 있을 때 확인 질문을 해야 하는 기준
- 외부 시스템에 쓰기 작업을 하기 전에 확인해야 하는 기준

## 모델과 fallback 설정

모델 설정에서는 기본 모델과 모델 파라미터를 지정합니다. Moldy 소스에는 `model_params`와 `model_fallback_ids`가 있으며, 런타임은 fallback 목록을 순서대로 해석합니다.

| 항목 | 의미 |
| --- | --- |
| 기본 모델 | 에이전트가 우선 호출하는 LLM |
| temperature | 답변 다양성 |
| top_p | 샘플링 범위 |
| max_tokens | 응답 최대 토큰 |
| fallback 모델 | 기본 모델 실패 시 시도할 대체 모델 목록 |

<Tip>
fallback 모델은 같은 제공자일 필요는 없지만, 각 모델을 호출할 수 있는 자격증명이 준비되어 있어야 합니다.
</Tip>

## 도구, MCP 도구, 스킬 연결

에이전트는 세 종류의 확장 요소를 가질 수 있습니다.

- **도구**: Moldy에 등록된 단일 작업입니다.
- **MCP 도구**: 사용자가 등록한 MCP 서버에서 발견한 도구입니다.
- **스킬**: 재사용 가능한 지식, 파일, 실행 패키지입니다.

에이전트 설정에서 연결한 항목만 해당 에이전트 런타임에 전달됩니다. MCP 서버를 등록해도 에이전트에 MCP 도구를 연결하지 않으면 그 에이전트는 해당 도구를 사용하지 않습니다.

## 서브에이전트

서브에이전트는 다른 에이전트를 현재 에이전트의 보조 실행 단위로 연결하는 기능입니다. 설정 화면은 자기 자신을 서브에이전트로 선택하지 못하도록 필터링하고, 중복 ID도 스키마 단계에서 거부합니다.

활용 예시는 다음과 같습니다.

- 조사 담당 에이전트와 요약 담당 에이전트를 분리
- 외부 도구 접근 권한이 필요한 작업을 별도 에이전트에 맡김
- 긴 작업을 작은 전문 작업으로 나누어 실행

## 미들웨어

미들웨어는 에이전트 런타임에 추가 동작을 끼워 넣는 설정입니다. 실제 API 스키마는 `middleware_configs` 배열을 받으며, 각 항목은 registry에 등록된 type만 허용합니다.

<Warning>
등록되지 않은 미들웨어 type은 서버에서 거부됩니다. 운영 중 새 미들웨어를 추가했다면 프론트엔드 선택 목록과 백엔드 registry가 함께 갱신되었는지 확인하세요.
</Warning>

## 오프너 질문

<img className="moldy-screenshot" src="/images/hancom/moldy/ko/agent-settings-opener.png" alt="에이전트 설정 오른쪽 패널의 오프너 질문 편집 화면" />

오프너는 사용자가 채팅을 시작하기 전에 볼 수 있는 추천 질문입니다. Moldy는 최대 12개의 질문을 허용하고, 각 질문은 비어 있으면 안 되며 길이 제한을 받습니다.

좋은 오프너는 사용자가 바로 실행할 수 있는 문장입니다.

```txt
이번 주 사용량을 요약해줘
최근 실패한 스케줄을 확인해줘
등록된 MCP 도구 중 사용할 수 있는 것을 알려줘
```

## Fix와 테스트 채팅

오른쪽 패널의 Fix는 에이전트 구성을 자연어로 조정하는 흐름입니다. 테스트 채팅은 저장 전 또는 저장 후 실제 응답을 확인하는 데 사용합니다.

1. 지침, 모델, 도구 구성을 바꿉니다.
2. 테스트 채팅에서 작은 요청을 보냅니다.
3. 도구 호출, 응답 형식, 비용/토큰 정보를 확인합니다.
4. 문제가 있으면 Fix 패널 또는 폼 설정에서 조정합니다.
5. 변경사항을 저장합니다.

## 스케줄 패널

<img className="moldy-screenshot" src="/images/hancom/moldy/ko/agent-settings-schedule.png" alt="에이전트 설정 오른쪽 패널의 스케줄 탭 화면" />

스케줄 탭은 해당 에이전트에 연결된 trigger를 보여주고 새 trigger를 추가합니다. 스케줄은 `one_time`, `interval`, `cron` 타입을 사용하며, 실행 결과를 기존 스케줄 대화방에 쌓을지 새 대화로 만들지도 설정할 수 있습니다.

## 메모리 설정

에이전트별 메모리 설정은 사용자의 전역 메모리 정책을 상속하거나 이 에이전트에서만 override할 수 있습니다. 민감하거나 임시적인 정보, 또는 에이전트 전용 정보만 다루는 경우 override를 사용하세요.

| 설정 | 값 |
| --- | --- |
| Memory policy | `inherit`, `off`, `ask`, `auto` |
| Scope policy | `inherit`, `agent_only`, `user_and_agent` |
| Trigger policy | `inherit`, `off`, `auto` |

전역 정책과 저장된 메모리 기록 흐름은 [메모리 관리](/hancom/moldy/ko/memory)를 참고하세요.

## API 패널

API 패널은 이 에이전트를 [Agent API](/hancom/moldy/ko/agent-api) 흐름과 연결합니다. 외부 API 배포에는 조건을 만족하는 에이전트와 서버용 API 키가 필요합니다. 에이전트가 배포 불가 상태라면 credential resolution 방식과 API 호출에 필요한 fixed-identity 조건을 만족하는지 확인하세요.

## 이미지와 설정 작업

<img className="moldy-screenshot" src="/images/hancom/moldy/ko/agent-settings-settings.png" alt="에이전트 아바타 이미지를 생성하거나 다시 생성하는 설정 패널" />

설정 탭은 에이전트 아바타 이미지를 생성하거나 다시 생성하는 작업을 제공합니다. 생성은 서버의 에이전트 이미지 API를 호출하며, 진행 중에는 로딩 상태와 진행 표시가 나타납니다.

## 비주얼 모드

<img className="moldy-screenshot" src="/images/hancom/moldy/ko/agent-settings-visual.png" alt="에이전트와 도구, 스킬, 스케줄 관계를 노드로 보여주는 비주얼 설정 모드" />

비주얼 모드는 같은 설정을 노드 기반으로 보여줍니다. 에이전트, 모델, 도구 상자, 스킬, 서브에이전트, 미들웨어, 스케줄 노드가 관계를 드러내므로 연결 누락이나 과도한 구성을 확인할 때 유용합니다.

## 삭제와 저장 주의사항

설정 화면은 변경된 상태를 감지하고 저장 여부를 구분합니다. 테스트 채팅은 동작 검증에 도움이 되지만, 일반 대화와 스케줄은 저장된 설정을 기준으로 실행됩니다.

삭제는 되돌릴 수 없는 작업이므로, 에이전트에 연결된 대화와 스케줄을 먼저 확인하세요. 활성 스케줄이 있는 에이전트는 삭제 전에 스케줄을 pause하거나 제거하세요.
