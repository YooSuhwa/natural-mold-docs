Moldy의 접근 모델은 사용자 소유 범위와 운영자 감독으로 구성됩니다. 대부분의 리소스는 인증된 사용자 소유 범위에서 조회되고, 운영자 기능은 `super_user` 권한으로 보호되며, 공개 접근은 read-only 대화 공유 링크로 제한됩니다.

이 문서는 Moldy 접근 제어를 설명하는 기준 문서입니다. 현재 Moldy 소스에는 세분화된 조직/워크스페이스 역할 정책이 없으므로, 프론트엔드 라우트, 백엔드 의존성, API 표면에서 확인되는 권한만 다룹니다.

자격증명, 메모리, 생성 파일, 공유 링크, Agent API key를 안전하게 운영하는 기준은 [보안과 프라이버시](/hancom/moldy/ko/security-privacy)를 참고하세요.

## 권한 모델

| 범위 | 설명 |
| --- | --- |
| 인증 사용자 | 자신의 에이전트, 대화, 자격증명, MCP 서버, 스킬, 메모리, 파일, 스케줄, 사용량, Agent API 키를 관리 |
| 운영자 `super_user` | 시스템 자격증명, System LLM, 모델 catalog mutation, 마켓플레이스 listing, 전체 활동 기록 관리 |
| 공개 방문자 | share token으로 공개된 대화 snapshot과 공유 파일만 읽기 |

실무 기준은 일반 애플리케이션 작업은 사용자 소유 범위에 머물고, 공용 플랫폼 설정은 운영자 소유 범위에 머무는 것입니다. 접근 문제를 점검할 때는 리소스가 없다고 판단하기 전에 현재 사용자, 리소스 소유자, `super_user` 필요 여부를 함께 확인하세요.

## 사용자 소유 리소스

에이전트, 대화, 사용자 자격증명, MCP 서버, 메모리 기록, 생성 파일, 스케줄, 배포, API 키는 현재 사용자 ID를 기준으로 조회됩니다. 존재하지 않는 리소스와 다른 사용자의 리소스는 같은 404 계열 응답으로 처리되어 리소스 존재 여부를 노출하지 않도록 설계되어 있습니다.

## 운영자 전용 기능

다음 기능은 `require_super_user`로 보호됩니다.

- 시스템 자격증명 목록, 생성, 수정, 삭제
- System LLM role slot 조회와 수정
- 모델 생성, 수정, 삭제, hidden model 포함 조회
- 마켓플레이스 public item의 listed 상태 변경
- k-skill sync 상태 확인
- 사용자 전체 활동 기록 확인

## 자격증명 보호와 감사

자격증명 응답은 복호화된 값을 반환하지 않습니다. 사용자 자격증명은 사용자 범위에서만 조회되고, 시스템 자격증명은 운영자 범위에서만 조회됩니다. 자격증명 수정/삭제 흐름은 감사 로그를 남기며, credential detail 화면에서 최근 audit log를 확인할 수 있습니다.

자격증명 감사 정보는 secret 값을 노출하지 않고 관리 변경 이력을 설명할 수 있어 지원 과정에 유용합니다. 공개 문서와 화면 캡처에는 자격증명 이름, 정의, 상태만 보여주고 원본 secret field는 포함하지 마세요.

## 공유 링크

공유 링크는 대화 소유자가 만든 read-only 공개 snapshot입니다. 공유된 메시지는 공개 조회 가능한 파일 artifact를 포함할 수 있습니다.

| 흐름 | 동작 |
| --- | --- |
| 공유 조회 | 현재 대화에 활성 share token이 있는지 확인 |
| 공유 생성 | 이미 활성 token이 있으면 같은 token을 반환 |
| 공유 회수 | 활성 token을 revoke하고 cache를 무효화 |
| 공개 조회 | 인증 없이 share token으로 snapshot, messages envelope, 공유 파일 조회 |

공개 조회는 revoke되었거나 알 수 없는 token에 대해 404를 반환합니다. 공개 endpoint는 IP 기준 rate limit과 짧은 cache를 사용합니다.

<Warning>
공유 링크는 로그인 없이 볼 수 있습니다. 공유 전 민감한 메시지, 파일, 도구 결과가 포함되어 있지 않은지 확인하세요.
</Warning>

## 관찰과 추적

대화 실행은 메시지, 도구 호출 결과, 생성 파일, usage, trace/debug 정보를 남깁니다. 에이전트 대화의 trace 페이지와 디버그 API는 실행 단계를 추적하는 데 사용됩니다. 스케줄 실행은 run status, error message, output preview, trace id를 기록합니다.

Agent API 호출은 API key actor로 귀속됩니다. 외부 실행을 조사할 때는 deployment public ID, API key prefix/last four, scope, run ID, audit outcome을 함께 확인하세요.
