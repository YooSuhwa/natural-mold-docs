에이전트 화면은 저장된 Moldy 에이전트를 실행하고 결과를 확인하는 화면입니다. 채팅, 대화 목록, draft conversation, 스트리밍 응답, 도구 결과, 생성 파일, 메모리 제안, 사용량 메타데이터, 공유 링크, trace, 설정 진입점을 함께 제공합니다.

메시지를 보내면 백엔드는 현재 에이전트 설정을 읽고 SSE 스트림으로 응답합니다. 이 문서는 에이전트가 이미 만들어진 뒤 사용자가 실제로 실행하는 흐름을 설명합니다.

<img className="moldy-screenshot" src="/images/hancom/moldy/ko/agent-chat.png" alt="Moldy 에이전트 채팅 화면" />

## 채팅 시작

1. 대시보드 또는 사이드바의 최근 에이전트에서 에이전트를 엽니다.
2. 기존 대화를 선택하거나 새 대화를 시작합니다.
3. 입력창에 요청을 입력합니다.
4. 응답, 도구 실행 결과, usage 정보를 확인합니다.
5. 필요한 경우 같은 대화에서 후속 요청을 보냅니다.

첫 테스트 메시지는 작고 평가하기 쉬운 요청으로 시작하세요. 응답 형식을 먼저 확인한 뒤 도구 호출, 파일, 스케줄, 긴 작업을 단계적으로 테스트하면 원인을 분리하기 쉽습니다.

사용자가 **새 대화**를 누르면 Moldy는 실제 대화 row를 만들기 전에 draft route를 보여줄 수 있습니다. 첫 메시지를 보내면 실제 conversation을 생성하고 URL을 새 conversation ID로 교체합니다. 사용자가 draft만 열고 떠나는 경우 빈 대화가 쌓이지 않도록 하기 위한 동작입니다.

## 메시지 스트리밍

Moldy는 `POST /api/conversations/{conversation_id}/messages` 요청에 대해 `text/event-stream`으로 응답합니다. 스트림은 메시지 조각, 도구 이벤트, 오류, message_end 이벤트를 포함할 수 있습니다.

스트리밍 중 연결이 끊기면 클라이언트는 run id 기반 stream 재연결 경로를 사용할 수 있습니다. 자세한 흐름은 [채팅 스트리밍 API](/hancom/moldy/ko/api-chat-streaming)를 참고하세요.

## 채팅 인라인 컴포넌트

Moldy 채팅은 단순 텍스트만 표시하지 않습니다. 메시지 본문, 도구 호출, 승인 요청, 생성 파일, 메모리 이벤트가 각각 전용 카드나 렌더러로 나타납니다. 스트리밍 중에는 부분 문법으로 깨질 수 있는 요소를 보수적으로 표시하고, 메시지가 완료되면 가능한 경우 풍부한 preview로 전환합니다.

카드별 의미, 사용자 행동, 실패 상태를 빠르게 찾으려면 [채팅 컴포넌트 갤러리](/hancom/moldy/ko/chat-components)를 참고하세요.

| 종류 | 화면에서 보이는 것 | 사용 시점 |
| --- | --- | --- |
| Markdown 본문 | 제목, 문단, 목록, 표, 링크, 인용문, 구분선 | 일반 답변과 설명형 응답 |
| 코드 블록 | 언어 라벨, 코드 영역, 완료 후 syntax highlight | 코드 생성, 로그, 설정 예시 |
| Mermaid | `mermaid` 코드 블록을 다이어그램으로 렌더링 | flowchart, sequence diagram, 구조 설명 |
| KaTeX 수식 | inline/display math 렌더링 | 계산식, 수학 설명, 모델 추론 근거 |
| 이미지 | 인라인 이미지, conversation image preview, 클릭 시 확대 보기 | 생성 이미지, 첨부 이미지, 도구 결과 이미지 |
| 검색 결과 카드 | 제목, URL, snippet, 외부 링크 | Tavily, Web, Naver Blog/News, Google Search/News |
| 파일 도구 카드 | 읽기, 쓰기, 수정 대상 파일명과 결과 | `read_file`, `write_file`, `edit_file` 실행 |
| 일반 도구 카드 | 도구 이름, 입력, 결과, 상태, 오른쪽 패널 확장 버튼 | 전용 UI가 없는 도구 실행 |
| 계획 카드 | todo 상태와 진행 중 항목 | `write_todos` 또는 계획형 실행 |
| 승인 카드 | 승인, 거절, 수정 JSON, 만료 countdown | tool approval, prompt approval, recommendation approval |
| 사용자 입력 카드 | 자유 입력, 단일/다중 선택, 단계형 질문 | `ask_user`, 명확화 질문, human-in-the-loop |
| 메모리 카드 | 메모리 제안, 편집, 승인, 거절, 저장 상태 | 사용자/에이전트 memory proposal과 save 이벤트 |
| 서브에이전트 카드 | 위임된 agent name과 실행 상태 | `task` 기반 sub-agent 실행 |
| Builder 카드 | phase timeline, 추천 승인, 이미지 선택/승인, draft config/approval | conversational builder 흐름 |

인라인 카드가 표시되면 카드 안의 상태가 최종 응답보다 더 구체적인 원인을 알려줄 수 있습니다. 예를 들어 모델 응답은 성공처럼 보여도 도구 카드에 credential 실패가 남아 있을 수 있고, 메모리 카드는 저장 전 사용자 승인을 기다릴 수 있습니다.

## 도구 결과 확인

도구가 호출되면 결과가 대화 안에 표시됩니다. 도구 결과를 볼 때는 다음을 확인합니다.

- 도구가 어떤 입력으로 호출되었는지
- 결과가 응답에 어떻게 반영되었는지
- 실패한 경우 credential, MCP server, model tool-calling 지원 중 무엇이 원인인지
- 승인 또는 resume이 필요한 실행인지

## 대화 관리

대화는 에이전트별로 저장됩니다. 대화 목록에서는 읽음 상태, 최근 사용 시각, 대화 제목, unread count를 확인할 수 있습니다. 스케줄 실행 결과도 대화와 연결될 수 있습니다.

Moldy는 메시지 edit, regenerate, branch switch, conversation read 처리 API를 제공합니다. 따라서 긴 대화에서 특정 지점부터 다시 시도하거나 다른 응답 branch를 선택할 수 있습니다.

대화 기록은 지원과 문제 분석의 근거이기도 합니다. 문제를 보고할 때는 권한이 허용하는 범위에서 대화, 메시지 branch, 도구 결과, trace를 함께 확인하세요.

## 파일, 메모리, 피드백

도구 실행은 file artifact를 만들 수 있습니다. 이 파일은 대화 안의 artifact card, 오른쪽 artifact 패널, [파일과 아티팩트](/hancom/moldy/ko/files-artifacts) 라이브러리에서 preview, favorite, download할 수 있습니다.

메모리가 활성화되어 있으면 실행이 저장된 메모리를 사용하거나 사용자 정책에 따라 메모리 제안을 만들 수 있습니다. 대화에 민감한 정보가 포함되어 있으면 승인 전에 proposal 내용을 검토하세요.

메시지 feedback은 feedback API로 기록하거나 지울 수 있습니다. 응답 품질을 평가할 때 feedback, trace, 파일, token usage를 함께 확인하세요.

## Trace 보기

대화별 trace와 debug trace는 실행 단계를 확인하는 데 사용합니다. trace 화면은 도구 호출, 모델 호출, 오류가 어느 단계에서 발생했는지 찾을 때 유용합니다.

## 공유 링크

공유 링크는 대화 snapshot을 로그인 없는 공개 보기로 제공합니다. 소유자는 공유 링크를 만들거나 회수할 수 있고, 공개 방문자는 share token으로 read-only snapshot을 봅니다. 공유 파일도 public share artifact endpoint를 통해 노출될 수 있습니다.

<Warning>
공유 링크는 인증 없이 접근됩니다. 공유 전에 민감한 메시지, 파일, 도구 결과가 포함되어 있지 않은지 확인하세요.
</Warning>
