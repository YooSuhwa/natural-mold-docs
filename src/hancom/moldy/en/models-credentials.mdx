---
title: Models and credentials
description: Understand user credentials, system credentials, model catalog entries, model tests, and fallback models.
---

Moldy separates model catalog entries from credentials. Models define which LLM can be selected and how it is displayed, while credentials provide the secrets required to call the provider.

This separation keeps model configuration auditable without exposing secret values. When a model call fails, verify the model catalog entry, the selected credential, and the runtime scope before changing agent instructions.

## User credentials

<img className="moldy-screenshot" src="/images/hancom/moldy/en/credentials.png" alt="Moldy user credentials screen" />

User credentials are managed from **Settings > Credentials** and are used by personal agents, tools, MCP servers, model tests, and model discovery. Backend responses do not return decrypted secret values; they expose status, field keys, OAuth capability, audit logs, and test results.

Supported credential definitions are registered under backend `credentials/definitions`. Current source includes OpenAI, Anthropic, Google GenAI, OpenRouter, Azure OpenAI, HTTP API key/basic/bearer, MCP OAuth2/secret, Google Workspace OAuth2, search, transportation, and public-data API definitions.

## System credentials

<img className="moldy-screenshot" src="/images/hancom/moldy/en/system-credentials.png" alt="Moldy system credentials screen" />

System credentials are operator-managed. They are stored with `is_system=true`, hidden from user pickers, and used by platform flows such as Builder, Assistant, image generation, and System LLM.

| Type | User credentials | System credentials |
| --- | --- | --- |
| Scope | Current user | Operator-managed |
| Read permission | Current user | `super_user` |
| Used for | User agents, tools, MCP servers | System LLM, Builder, Assistant, image generation |
| UI | Credentials | System Credentials |

## Model catalog

<img className="moldy-screenshot" src="/images/hancom/moldy/en/models.png" alt="Moldy model catalog screen" />

**Settings > Models** manages provider, model name, display name, base URL, pricing, context window, modality, function calling, reasoning, visibility, default credential metadata, health status, and benchmark ranking data when available.

Model create, update, and delete operations require operator permission. Delete is refused when any agent still uses the model.

Model metadata affects runtime choices, reporting, and catalog sorting. Pricing metadata is used for estimated cost, feature flags affect tool-calling and display decisions, rankings can help compare known public models, and visibility controls which models users can select.

## Model testing and discovery

A model entry is not enough by itself. Moldy needs a usable credential. The backend supports registered model tests, preview tests, credential-based model discovery, and health checks.

When a test fails:

1. Check provider and model name.
2. Confirm the model has a default credential or the test request specifies one.
3. Check base URL for providers that require it.
4. Remember that pricing and feature flags affect display and cost calculation, not just callability.
5. For discovered models, review benchmark rankings and missing ranking fallbacks before saving.

## Agent model selection and fallback

Agents can have a primary model and an ordered fallback model list. Chat runtime tries the primary model first and resolves fallback entries in order.

Fallback is a resilience mechanism, not a permission bypass. Each fallback model still needs a usable credential in the correct scope and should be tested before relying on it in production agent workflows.

<Note>
Fallback models still need usable user credentials for their providers. System credentials are not a general fallback for user chat runtime.
</Note>
